# Skill Eval Loop

The slide skill should improve through examples, not vibes.

## Loop

1. Draft or revise the skill.
2. Save realistic prompts in `evals/evals.json`.
3. Run each prompt with the skill and without the skill, or against the previous skill version.
4. Save outputs under a real eval run folder, such as `evals/runs/iteration-N/`.
5. Grade objective checks where possible.
6. Ask the human to review qualitative output.
7. Compare quality, token use, and time.
8. Revise the skill based on recurring failures.
9. Expand the eval set once the first examples pass.

## Promotion Protocol

Use this when deck review reveals a reusable failure.

1. Write the failure in the deck's `qa/` folder.
2. Add a row to `docs/heuristic-registry.md`.
3. Choose the smallest reusable target: skill reference, maintainer doc, validator, template, design contract, example, or eval.
4. Implement that target without rewriting unrelated workflow.
5. Prove it with a validator, browser QA, committed example, eval, or captured review artifact.
6. Mark the registry row as `tested` only after the proof exists.
7. Compress repeated narrow rules into one clearer rule when the pattern stabilizes.

This is the repo's heuristic-learning loop: absorb review feedback, repair the system, test the repair, then compress the lesson so future decks improve without adding prompt clutter forever.

## What To Measure

- Story coherence from title-only outline.
- Claim traceability.
- Correct handling of research mode.
- Diagram accuracy from code or source.
- Visual aid quality.
- Speaker note naturalness.
- Token usage for whole-deck vs slide-local repairs.
- Whether repair runs changed only the targeted artifact.
- Token usage and duration for each run.
- Whether the skill improves over a captured baseline prompt without the skill.

## Baseline Comparison

For each realistic eval, keep two outputs when possible:

- `with_skill`: follows `skills/slide-generator/SKILL.md`.
- `without_skill`: same prompt with no slide-generator guidance.

Compare both qualitatively and with objective checks. The skill should justify its token cost by producing better planning artifacts, stronger accuracy discipline, and fewer generic slides. A hand-authored baseline is useful for regression testing, but it does not prove the skill beats a real one-shot model run.

## Source-Backed Eval

The first source-backed eval project is:

```txt
evals/source-backed/hackathon-rubric-eval/
```

Run it with:

```bash
npm run deck:build -- evals/source-backed/hackathon-rubric-eval --render
npm run deck:build -- evals/source-backed/hackathon-rubric-eval/baseline-one-shot --render
npm run eval:ab -- evals/source-backed/hackathon-rubric-eval
```

The rendered deck, QA output, and `qa/ab-eval-report.json` are generated by the commands above and are intentionally not committed. This eval checks whether the skill can turn reputable presentation guidance into a concise hackathon/demo deck without inventing project-specific results. Review it against:

- rubric alignment,
- one-slide-per-minute pressure,
- source-backed claims,
- demo path and fallback,
- audience questions and objections,
- technical proof boundaries,
- slide-by-slide clarity,
- whether the skill output exceeds the frozen structural proxy baseline on deterministic proxy signals.

This is the first comparative eval fixture. It is not proof that the skill beats one-shot generation: the baseline is self-authored, intentionally generic, and should be replaced with captured model-specific one-shot outputs for serious model studies. The scorer checks deterministic proxies such as hard gates, claim traceability, rubric fit, demo proof, browser QA, timing, visual diversity, and review-loop evidence. It does not semantically verify that each external URL supports each claim; that requires a future source-audit artifact with evidence anchors.

## Captured One-Shot Protocol

Use this when the claim is "the skill beats one-shot generation" rather than "the structural proxy did not regress."

1. Use the same source pack and user prompt.
2. Run the target model without the slide-generator skill, repo references, or artifact contract.
3. Save the raw model output under a captured baseline folder.
4. Convert only what is required to render and validate the deck. Do not add missing rubric slides, audience objections, proof visuals, claim IDs, or repair-loop evidence unless they appeared in the raw output.
5. Render and QA both decks with the same scripts.
6. Run `npm run eval:ab -- <eval-root> --baseline <captured-baseline-dir>`.
7. Pair the proxy score with blind screenshot review and semantic source audit before claiming a real quality win.

## Eval Viewer

When an eval-viewer tool or script is available, generate a review page before revising the skill. The human should see outputs side by side and leave feedback. The feedback should drive the next iteration.

If no viewer is available, keep the same structure on disk and report concise links to the generated artifacts.
