# Planning Mode

Use Planning Mode before rendering when the user wants a high-stakes deck, asks for a plan, is unsure about theme or flow, wants research-backed persuasion, or is disappointed with a shallow draft.

Planning Mode is not a pause forever. It is a short contract that prevents the system from making beautiful but weak slides.

## Output

Create `work/deck-plan.md` or update `work/intake-brief.md` with:

- deck job, audience, presenter, target next action, talk length, and output format,
- one-sentence promise,
- audience shift: from what they believe now to what they should believe or do after,
- deck route: mode, hook, proof spine, main deck count, appendix policy,
- research plan: what must be sourced, what can be inferred, what needs current web checks,
- slide logic table: title, slide job, proof object, likely audience question, and next action,
- design directions: 2 or 3 theme options with tradeoffs when taste is unclear,
- QA plan: claim audit, story audit, visual QA, mobile/export QA, and final humanizer pass,
- assumptions and open questions that change the answer.

## Slide Logic

Every planned slide needs:

- one job,
- one audience question it answers,
- one proof object: number, table, chart, source excerpt, screenshot, diagram, demo, quote, or decision gate,
- one takeaway title,
- one reason it belongs in the main deck rather than appendix.

Main deck persuades or teaches. Appendix defends. If a table is important for trust but too dense for the story, put a summary in the main deck and the full table in appendix.

## Communication Rules

Use public-talk clarity without losing evidence:

- Start with the decision, benefit, user, or question. Do not open with broad background.
- Make the audience care before showing mechanism.
- Turn research into "so what", not only "what exists".
- Replace vague steps with actor, first action, artifact, and success signal.
- Use plain definitions for terms such as pilot, waitlist, lead, soft hold, utilization, or approved use.
- Repeat the core idea in the title, visual, notes, and close.
- Keep speaker notes speakable; do not paste research paragraphs.

## Reference Patterns

Borrow patterns, not assets:

- TED/public-speaking style: one clear idea, concrete example early, short story only when it creates stakes, memorable close.
- Consulting style: answer first, issue tree or criteria, evidence exhibit, decision close.
- Gamma/Plus/SlideSpeak style: outline before render, source-to-deck adapters, global edits separate from slide-local edits.
- Canva/Figma style: theme directions, visual cover/section energy, image and asset breadth, but no fake evidence or unsourced polish.

## Planning Gate

Do not render until these are true:

- The title-only story makes sense.
- The strongest objection is answered in the main deck or named as appendix/Q&A.
- Every factual slide has claim IDs or explicit research tasks.
- The theme direction is chosen or a recorded default exists.
- Main slides fit the talk length.
- Dense material has a table, chart, appendix, or speaker-note home.

If the plan fails, repair the plan before rendering.
